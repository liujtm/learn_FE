# React 逐行带读

这份文档配合 `src/main.jsx` 和 `src/App.jsx` 一起看。

如果你是后端开发，先抓住一个总模型：

- `main.jsx` 负责启动应用
- `App.jsx` 负责根据当前状态生成页面
- 用户交互通过事件进入
- `setState` 会触发组件重新执行
- React 根据新旧结果对比后更新页面

## 1. 从入口开始：`src/main.jsx`

```jsx
import { StrictMode } from 'react'
import { createRoot } from 'react-dom/client'
import './index.css'
import App from './App.jsx'
```

这几行各自负责：

- `StrictMode`
  React 在开发环境下的一个检查容器，帮助你尽早发现不规范写法
- `createRoot`
  React 18 的挂载入口
- `./index.css`
  全局样式
- `App`
  当前页面的根组件

继续往下：

```jsx
createRoot(document.getElementById('root')).render(
  <StrictMode>
    <App />
  </StrictMode>,
)
```

这段代码可以拆成两步理解：

1. 先从真实 DOM 里找到 `index.html` 里的 `#root`
2. 再把 React 组件树挂载进去

你可以把它类比成：

- `index.html` 提供一个挂载点
- React 在这个挂载点里“接管”后续 UI 渲染

## 2. 再看主组件：`src/App.jsx`

最开头：

```jsx
import { useEffect, useMemo, useState } from 'react'
import './App.css'
```

这里是组件依赖声明。

- `useState`
  保存组件本地状态
- `useEffect`
  处理副作用
- `useMemo`
  缓存派生值

### 2.1 静态配置区

```jsx
const lessons = [...]
```

这部分不是状态，而是常量配置。

这是前端里很常见的写法：

- 固定展示内容放常量
- 会变化的内容放 state

你读代码时，先分清这两类，脑子会轻很多。

### 2.2 状态定义区

```jsx
const [count, setCount] = useState(0)
const [step, setStep] = useState(1)
const [name, setName] = useState('后端同学')
const [logs, setLogs] = useState([])
```

这是整个组件最核心的一段。

每一行都表示：

- 左边第一个变量是“当前值”
- 左边第二个变量是“更新函数”
- `useState(...)` 里的值是“初始值”

例如：

```jsx
const [count, setCount] = useState(0)
```

意思就是：

- `count` 当前初始值是 `0`
- 调用 `setCount(...)` 可以更新它
- 更新之后 React 会重新执行 `App()`

这和后端里“修改普通局部变量”的感觉完全不同。
React 不能靠你直接写：

```jsx
count = count + 1
```

来刷新页面。
它必须通过 `setCount(...)` 知道“状态变了”。

### 2.3 副作用区：`useEffect`

```jsx
useEffect(() => {
  setLogs((prev) => [
    `count 从 ${count - step} 变成 ${count}，组件在 ${new Date().toLocaleTimeString()} 重新渲染`,
    ...prev,
  ].slice(0, 5))
}, [count, step])
```

这一段非常适合拿来建立对 `effect` 的第一印象。

它表达的是：

- 当 `count` 或 `step` 发生变化后
- React 完成这次渲染提交
- 然后执行这里的逻辑

依赖数组：

```jsx
[count, step]
```

表示只有这两个值变了，`effect` 才重新运行。

这在工程里通常会被用来做：

- 发请求
- 订阅/取消订阅
- 写日志
- 操作定时器

这里我们故意只做日志，是为了让链路更好看清。

### 2.4 派生值区：`useMemo`

```jsx
const message = useMemo(() => {
  if (count === 0) return '还没开始交互，先点一下按钮。'
  if (count < 5) return '这就是最基础的状态更新。'
  return '你已经触发了多次渲染，可以开始看 effect 和 props 了。'
}, [count])
```

这里的重点是：

- `message` 不是独立状态
- 它是由 `count` 推导出来的

所以你不需要：

- 再单独来一个 `messageState`
- 再自己同步更新它

你只需要描述“它怎么从 `count` 算出来”，React 就能在依赖变化时重新计算。

这个思想对后端也很有帮助：

- 原始数据是一类
- 派生数据是一类
- 不要把能推导出来的东西重复存储

### 2.5 返回 JSX

```jsx
return (
  <main className="page">
    ...
  </main>
)
```

你可以把 JSX 理解成：

- 长得像 HTML
- 但其实是 JavaScript 里的 UI 描述

React 每次执行 `App()`，都会重新得到一份 JSX 结果。

然后 React 会做两件事：

1. 拿这次结果和上次结果比较
2. 只更新真正有变化的 DOM 部分

### 2.6 受控输入框

```jsx
<input value={name} onChange={(e) => setName(e.target.value)} />
```

这是 React 表单最经典的一种写法。

这意味着：

- 输入框显示什么，由 `name` 决定
- 用户输入变化后，通过 `onChange` 把新值写回 `name`

这类组件叫“受控组件”。

链路是：

1. 页面展示 `name`
2. 用户输入
3. 触发 `onChange`
4. 调用 `setName`
5. `App()` 重新执行
6. 输入框拿到新的 `value`

### 2.7 按钮点击事件

```jsx
<button type="button" onClick={() => setCount((v) => v + step)}>
  +{step}
</button>
```

这行非常值得你反复看。

发生的事情是：

1. 用户点击按钮
2. 触发 `onClick`
3. 执行 `setCount((v) => v + step)`
4. React 记录状态变化
5. 组件重新执行
6. 页面里的 `count`、`message`、日志区域都可能变化

这里用了函数式更新：

```jsx
setCount((v) => v + step)
```

好处是它基于“上一次最新值”来更新，能避免一部分闭包和异步批处理带来的问题。

### 2.8 列表渲染

```jsx
{lessons.map((lesson) => (
  <li key={lesson.title}>
```

React 渲染列表最常见就是 `map(...)`。

这里的 `key` 很重要。

它是 React 用来识别列表项身份的。
如果以后你做动态列表、增删改、排序，`key` 的选择会直接影响更新行为。

## 3. 一次完整交互的运行链路

假设你打开页面后点了一次 `+1`：

1. 浏览器触发按钮点击事件
2. 执行 `setCount((v) => v + step)`
3. React 安排当前组件更新
4. React 重新调用 `App()`
5. 重新读取 `count`、`step`、`name`、`logs`
6. 重新计算 `message`
7. 重新生成 JSX
8. React 对比前后结果
9. 更新页面中变化的文本节点
10. 因为 `count` 变了，`useEffect` 再执行一次
11. `setLogs(...)` 触发新一轮更新
12. 日志列表刷新

这里有个很重要的感觉：

React 不是你点一下按钮就“直接改页面”。
它更像是：

- 先改状态
- 再重新算 UI
- 最后同步到页面

## 4. 你调试时最该打的日志

推荐加在 3 个地方：

### 4.1 组件函数顶部

```jsx
console.log('render App', { count, step, name, logs })
```

帮助你观察什么时候重新渲染。

### 4.2 按钮点击里

```jsx
onClick={() => {
  console.log('click +step')
  setCount((v) => v + step)
}}
```

帮助你看事件入口。

### 4.3 `useEffect` 里

```jsx
useEffect(() => {
  console.log('effect run', { count, step })
}, [count, step])
```

帮助你看副作用什么时候发生。

## 5. 读完后建议你马上做的练习

1. 给 `count` 增加一个最小值 0
2. 把 `reset` 改成同时清空日志
3. 提取一个 `CounterPanel` 子组件
4. 增加一个异步“模拟请求”按钮
5. 再加一个 `useEffect`，观察多个 effect 的执行感受

做完这些，你对 React 的“状态驱动视图”就会有比较扎实的第一感觉。
